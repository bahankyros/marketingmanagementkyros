# V2 Roadmap

## Summary
Build the Marketing Connector V2 upgrade in backend-first order: freeze schemas, migrate user role and outlet identity, secure Firestore rules, implement sales import and derived monthly budgets, then layer in outlet-scoped field operations, calendars, and real-time notifications. No React feature build should begin until the schema contract, rules contract, and migration path are stable.

## Locked Decisions
- Sales storage model: `sales` stores one document per `outlet_id + month_key`.
- Budget storage model: `budgets` stores one locked company-level monthly rollup.
- `budget_rate` is fixed at `0.02`.
- `marketing_budget_total` is always derived, never hand-entered.
- `outlet_id` is the canonical supervisor authorization key.
- `outlet_name` is stored as denormalized display data.
- `mascot_bookings` becomes the booking request and approval source of truth.
- `notifications` should be generated by trusted backend logic, not only by client UI flows.
- Legacy fields like `users.outlet` and `mall_displays.outlet` must be migrated or mapped before strict V2 rules go live.

## Collection Contract
- `users/{auth_uid}`
- `sales/{month_key}__{outlet_id}`
- `budgets/{month_key}`
- `vouchers/{autoId}`
- `mall_displays/{slot_id}`
- `tasks/{autoId}`
- `events/{autoId}`
- `mascot_bookings/{autoId}`
- `notifications/{autoId}`

## Phase 0 — Contract Freeze
- [ ] Freeze the V2 collection list and document ID strategy.
- [ ] Freeze canonical outlet identity fields: `outlet_id` and `outlet_name`.
- [ ] Freeze enum values for roles, task statuses, mascot booking statuses, mascot status, and notification types.
- [ ] Freeze CSV import header names and month format as `YYYY-MM`.
- [ ] Freeze the rule that budgets are derived from imported sales only.

## Phase 1 — Identity And Settings Foundation
- [ ] Expand `users` documents to include `outlet_id`, `outlet_name`, and `status`.
- [ ] Backfill all existing supervisor users with a real `outlet_id`.
- [ ] Normalize admin and finance users to empty-string outlet fields.
- [ ] Define the admin user-management sync contract between Firebase Auth and Firestore `users`.
- [ ] Define the outlet master mapping table or constant so imports and supervisor scoping use the same IDs.

## Phase 2 — Firestore Rules And Storage Contract
- [ ] Rewrite `firestore.rules` helpers to read both role and `outlet_id` from `users/{uid}`.
- [ ] Enforce that supervisors can only read and write documents whose `outlet_id` matches their own.
- [ ] Lock `events` to admin CRUD and non-admin read-only.
- [ ] Lock `mall_displays` supervisor updates to their own outlet only.
- [ ] Lock `tasks` so admins create and assign while supervisors only update progress and proof fields for tasks assigned to them.
- [ ] Lock `vouchers` so supervisors can only log claims for their own outlet.
- [ ] Lock `mascot_bookings` so supervisors can create requests for their own outlet while admins alone approve and assign.
- [ ] Define Firebase Storage path rules for proof-image uploads before any upload UI is built.

## Phase 3 — Sales Import And Budget Engine
- [ ] Define the downloadable CSV or Excel template columns exactly:
  - `month_key`
  - `outlet_id`
  - `outlet_name`
  - `total_sales`
  - `grab_gross_order_value`
  - `grab_commission_fees`
  - `grab_ad_spend`
  - `foodpanda_gross_order_value`
  - `foodpanda_commission_fees`
  - `foodpanda_ad_spend`
- [ ] Build a parser and validator contract that rejects the whole file if any required header or row is invalid.
- [ ] Normalize all currency fields to numeric MYR values before write.
- [ ] Calculate delivery net profit during ingestion:
  - `net_profit = gross_order_value - commission_fees - ad_spend`
- [ ] Write one sanitized `sales` document per outlet-month.
- [ ] After a successful import batch, derive or overwrite `budgets/{month_key}` for the current month.
- [ ] Lock the budget doc for that month by writing:
  - `sales_rollup_total`
  - `budget_rate`
  - `marketing_budget_total`
  - `locked = true`
  - `lockedAt`
- [ ] Preserve import auditability with `source_file_name`, `source_batch_id`, `imported_by_uid`, and timestamps.

## Phase 4 — Field Ops Data Layer
- [ ] Extend `mall_displays` to carry canonical `outlet_id`, `outlet_name`, and stable `slot_code`.
- [ ] Migrate legacy mall display docs that only use `outlet`.
- [ ] Create the `vouchers` collection with supervisor outlet identity auto-captured from auth.
- [ ] Create the `tasks` collection with optional event linkage through `event_id`.
- [ ] Define the task lifecycle:
  - admin creates task
  - supervisor marks `in_progress`
  - supervisor submits `proof_text` and `proof_image_*`
  - admin reviews to `approved`, `rejected`, or `completed`
- [ ] Keep proof upload state separate from Firestore document state so incomplete uploads never write partial proof fields.

## Phase 5 — Logistics And Calendar Layer
- [ ] Refactor the event model to a true calendar shape with `startAt` and `endAt`.
- [ ] Treat `events` as admin-owned records and read-only for everyone else.
- [ ] Use `tasks.event_id` as the task/calendar integration point instead of embedding tasks inside event documents.
- [ ] Create `mascot_bookings` as the mascot request-and-approval calendar source of truth.
- [ ] Define mascot booking flow:
  - supervisor creates `pending` request
  - admin approves or rejects
  - admin assigns handler, final mascot status, and location
  - completed bookings remain in history for audit and scheduling
- [ ] If existing `mascot_schedule` and `mascot_logs` data must remain, define a migration or coexistence plan before reads switch to `mascot_bookings`.

## Phase 6 — Notifications Backbone
- [ ] Create the `notifications` collection and freeze its enum contract.
- [ ] Implement trusted notification creation for:
  - task assigned -> notify assigned supervisor
  - task marked proof submitted or completed -> notify admin
  - mascot booking requested -> notify admin
  - mascot booking approved or rejected -> notify requesting supervisor
- [ ] Ensure notifications are generated by trusted backend logic so they cannot be spoofed or skipped by the client.
- [ ] Keep notification documents small and reference the source entity by `entity_type` and `entity_id`.

## Phase 7 — Read APIs, Hooks, And Query Strategy
- [ ] Add role-aware query helpers before building any new screen.
- [ ] Require supervisor reads on outlet-bound collections to query by `outlet_id`.
- [ ] Keep admin financial queries month-bounded and index-friendly.
- [ ] Build shared selectors or helpers for:
  - budget derivation
  - delivery net-profit derivation
  - monthly chart rollups
  - unread notification counts
- [ ] Do not allow finance math to live directly in JSX.

## Phase 8 — UI Build Order
- [ ] Build Admin User Management after the Auth sync contract is stable.
- [ ] Build Sales Import and Budget History after the CSV parser and budget rollup are proven.
- [ ] Build Delivery Analytics after `sales.delivery.*` data is populated reliably.
- [ ] Build supervisor voucher logging after outlet-scoped auth is verified.
- [ ] Build mall display editing after `mall_displays` outlet migration is complete.
- [ ] Build tasks and proof submission after storage paths and rules are in place.
- [ ] Build event calendar after the event schema migration is complete.
- [ ] Build mascot booking UI after `mascot_bookings` approval flow is locked.
- [ ] Build notifications UI last, once notification creation is already trusted and real-time.

## Test Checklist
- [ ] User provisioning keeps Firebase Auth and Firestore `users` in sync.
- [ ] A supervisor for one outlet cannot read or write `mall_displays`, `tasks`, `vouchers`, or `mascot_bookings` for any other outlet.
- [ ] CSV validation rejects missing headers, malformed months, blank sales values, and unknown outlet IDs.
- [ ] Uploaded sales roll up to exactly `2%` in the locked monthly `budgets` document.
- [ ] Grab and Foodpanda net profit matches the stored formula.
- [ ] Task assignment, proof submission, and admin review all respect status transitions and required fields.
- [ ] Event calendar allows admin CRUD and non-admin read-only access.
- [ ] Mascot booking allows supervisor request creation only for their own outlet and admin approval afterward.
- [ ] Task assignment and task completion each create one notification for the correct recipient.
- [ ] Legacy `users.outlet` and `mall_displays.outlet` data are backfilled before strict outlet-based rules go live.

## Assumptions
- `outlet_id` becomes the only trusted authorization key for outlet-scoped data.
- Historical graphs will read from monthly rollups instead of unbounded raw scans.
- Finance remains in the system but does not own any new write flows in this V2 scope.
- Existing `mascot_logs` may remain as historical data until a separate cleanup pass archives or replaces them.
